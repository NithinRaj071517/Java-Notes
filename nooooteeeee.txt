Waits:
------
	Definition:
		Waits in Selenium are used to synchronize the WebDriver with the application, ensuring that elements are loaded or visible before interaction, instead of failing immediately.
		
	Need:
		We need waits in automation testing because modern web applications are dynamic and elements may load due to AJAX calls, page refreshes, or delayed responses. Without waits, Selenium may try to interact with elements that are not yet available.
	Types:
		1. Implicit Wait
			Global wait applied to all elements
			WebDriver waits up to the specified time before throwing NoSuchElementException
			Defined once and applied throughout the driver session
			
		2. Explicit wait
			Applied to specific element Wiats for a  specific condition.
			Uses WebDriverWait with ExpectedConditions
				WebDriverWait wait = new WebDriverWait(driver, Duration.ofSeconds(20));
				wait.until(ExpectedConditions.visibilityOf(element));
			
		3. Fluent wait
			Similar to explicit wait with polling frequency and Exception ignoring
			
			
	Although Selenium allows using both waits together and it does not throw any exception, it is not recommended because implicit wait gets added to explicit wait internally, resulting in unpredictable and increased execution time.Hence, in real-time automation frameworks, we usually disable implicit wait and rely on explicit waits for better control.

ArrayList and LinkedList:
------------------------		

	Definition:
		ArrayList and LinkedList are implementation classes of the List interface used to store ordered and dynamic collections of elements.

	ArrayList:
		1. It is an dynamic array
		2. Its access time is fast
		3. Insertion/Deletion is slow
		4. Uses a resizable array
		5. When capacity is full → creates a new array (approx 1.5x size) and copies elements
		6. It follows the insertion order and allows duplicates
	 

	LinkedList:
		1. It is doubly linked list(node based)
		2. Its access time is slow 
		3. Insertion/Deletion is fast 
		4. Uses nodes (each node has data + next + previous reference)
		5. No shifting required during insert/delete
		6. It follows the insertion order and allows duplicates

	HashSet:
		1. Stores only value
		2. Insertion order not maintained 
		3. Duplicates are not allowed 
		4. Allows to add one null value
		5. Uses HashMap internally
	 

	HashMap:
		1. Stores  key value pair representation 
		2. Insertion order not maintained 
		3. Duplicates Keys are not allowed but duplicate values allowed 
		4. Allows one null value in key & multiple null values
		5. HashMap works based on hashcode() and equals() for fast retrieval

	HashSet is used in window handling for unique window ID
	and HashMap used in test data assigning like user name & password

Locators:
--------
	Definition:
		Locator is used to identify and locate web elements on a web page so that Selenium can perform actions like click, sendKeys, or getText.
		Since web applications are built using HTML, locators help Selenium uniquely identify elements from the DOM.
		
	Types:
		1. Basic Locators 
				Id
				Name
				ClassName
				TagName
		2. Advanced Locators
				Link text
				Partial link text
				CSS selector
				Xpath
	
	Most preferred locator in Selenium:
		1.By.id because:
			IDs are unique in the DOM
			They are fastest for element identification
			They make scripts more stable and readable
		2. If Id is not available next preferred
			Name
			css selector
			Relative Xpath
			
	Less preferred locator in Selenium:
		1. By.classname:
			Same class can be used for multiple elements, which may lead to incorrect element identification if not handled properly.
		2. Absolute xpath:
			It is tightly coupled to the DOM structure and breaks easily when UI changes.
	
	CSS vs Xpath:
		1. CSS:
			CSS Selector uses CSS syntax to locate elements based on attributes such as id, class, tag name, and relationships.
			Syntax simpler
			DOM traversal forward only
			Performance is faster
			
		2. Xpath:
			XPath uses XML path expressions to locate elements by traversing the DOM structure.
			Syntax more powerful
			DOM traversal both backward & forward 
			Performance is slightly slower
			
		Preferred CSS because it is faster , cleaner and easy to maintain. 
		However , Xpath is unavoidable when 
			Locating elements using text
			Traversing from child to parent
			Handling complex dynamic elements
			
			
String:
------
	Definition:
		String in Java 	is an object that represents sequence of characters, implemented using String class from java.lang package.
		Strings are immutable, meaning once created, their value cannot be changed. If any modification is performed, a new object is created instead of modifying the existing one.
		
	Memory allocation:
		String literals are stored in the String Constant Pool (SCP) inside the heap. If the same value already exists, JVM reuses the reference to optimize memory.
		
		String objects (new keyword) are always created in the heap memory, even if the value already exists.
			
			String s1 = "Test";
			String s2 = "Test";  // same reference from SCP
			String s3 = new String("Test"); // new object in heap
			
	Reason for immutability:
		Security (important for URLs, DB connections)
		Caching (String pool reuse)
		Thread safety (no synchronization required)		
		
		

String vs String Buffer vs String Builder:
-----------------------------------------
	Definition:
		String, StringBuffer, and StringBuilder are classes used to represent sequences of characters in Java, but they differ in mutability, thread safety, and performance.
		
	
	Difference:
		1. String:
			Immutable
			Thread safe
			Stored in Sting Constant Pool
			
		2. String Buffer:
			Mutable
			Thread safe
			Slower due to synchronization overhead.
		
		3. String Builder:
			Mutable
			Not Thread safe
			Faster due to no synchronization .

Frames:
-------
	Definition:
		Frames / iFrames are HTML elements that embed another web page inside a parent web page.
		Each frame has its own DOM, and Selenium can interact only with the currently focused DOM.
		Therefore, before interacting with elements inside a frame, we must switch Selenium’s control to that frame.
		
	Methods to access Frames:
		1, Switching frame Using index
			driver.switchTo().frame(0);
						
		2. Switching frame Using String (String may be Name or Attribute)
			driver.switchTo().frame("Name");
		
		3. Switching frame Using WebElement
			driver.switchTo().frame(frameElement);
			
	parentFrame() --> is used to switch control from the current frame to its immediate parent frame.

	defaultContent() --> is used to switch control back to the main page, i.e., outside all frames, regardless of the nesting level.
	
	Nested Frames:
		Nested frames are handled by switching sequentially from the main page to the parent frame and then to the child frame. Selenium does not support jumping directly to an inner frame. To exit, parentFrame() moves one level up, while defaultContent() returns to the main page.

API (Application Programming Interface):
---------------------------------------
	Definition:
		API is an set of rules and protocols that allows to  communicate from one application with another application.
		It works on request - response mechanism where one system sends request and another system returns response.
		
	Examples:
		When you use a mobile app to book a ticket, the app communicates with the server through APIs to fetch available tickets and confirm the booking.
		
	API Testing:
		API Testing is a type of testing where we validate the functionality, reliability, performance, and security of APIs without involving the UI.
		It involves sending requests to the API and verifying the responses such as status codes, response body, headers, and data correctness.
		
	API testing difference from UI testing:
		API testing is faster and more reliable because it doesn’t depend on UI elements
		API testing helps identify issues early in development, while UI testing is usually done later
		API Testing is done at the backend level, whereas UI testing is done at the front-end level
		
	Real-time example: Instead of testing a login through UI, we directly send login API request and validate the response token.
	
	Types :
		1. Based on Access level
				Public API (Open APIs) --> Available for anyone to use (e.g., Google Maps API)
				Private API (Internal APIs) --> Used within an organization for internal communication
				Partner API --> Shared with specific business partners with controlled access
				Composite API --> Combine multiple APIs into a single request (used to improve performance)
		2. Based on Architecture level
				REST API --> Uses HTTP methods (GET, POST, PUT, DELETE), lightweight, widely used
				SOAP API --> Uses XML, more secure, follows strict protocol.
				
				
	API Request Components:
		An API request is sent by the client to the server to perform an action.
		It contains:
			HTTP Method (GET, POST, PUT, DELETE)
			URL (Endpoint) → identifies resource
			Headers → metadata (Authorization, Content-Type)
			Path Parameters → part of URL (/users/101)
			Query Parameters → filters (?status=active)
			Request Body → data sent (mainly in POST/PUT)
			Authentication → token, API key, etc.
		
		
	API Response Components:
		An API response is sent by the server back to the client after processing the request.
		It contains:
			Status Code → (200, 201, 400, 404, 500)
			Response Body → actual data (JSON/XML)
			Headers → metadata
			Response Message → success/error message	
			
			
	HTTPs status code are the  part of API response that indicate the result of the request sent by the client to the server
	It has different categories such as 
		1xx--> Request was received and the server is continuing the process.
		2xx--> Indicates the success reason 
				201 - Resource Created
				200 - Request was successful 
		3xx--> Indicates redirection
			
		4xx--> Error on the client side
				400 - Bad request 
				401 - Unauthorized
				403 - forbidden
		5xx--> Error on the server side. 
				500 - Internal Error
				501 - Bad Gateway

Encapsulation:
--------------
	Definition:
		Encapsulatin is an OOP concept where data and methods are wrapped together into a single unit and internal data is hidden from outside access.

	Uses:
		To achieve data hiding 
		To control how data is accessed or modified
		To improve security, maintainability, and flexibility.
		
	Internal working:
		Variables are declared as private
		Access is provided through public getter and setter methods
		We can also add validation logic inside setters to control incorrect data
		
	Real-time Project Example:
		In Selenium Page Object Model (POM), we declare WebElements as private and expose them via public methods (actions like click, sendKeys).
		This ensures:
			Elements are not directly accessed outside the class
			Changes in locators don’t affect test scripts
			Code becomes clean, reusable, and maintainable
			
			
	How Encapsulation Provides Data Security:
		Encapsulation ensures data security by restricting direct access to variables and allowing access only through controlled methods (getters & setters).
		This helps in:
			Preventing unauthorized modification
			Adding validation before updating data
			Controlling business rules centrally
			
	Real-world Scenario (Banking System):
		In a banking application, the account balance should not be modified directly.		
		User cannot directly change balance
		Only allowed operations (deposit/withdraw) are exposed
		Invalid operations are restricted using validation logic

Polymorphism:
------------
	Definition:
		Polymorphism means “one name, many forms”, where the same method behaves differently based on input parameters or object type.

	Types:
		1. Compile Time Polymorphism (Static Binding)
			Achieved using Method overloading
			Same method name with different method signatures:
				Different number of parameters
				Different data types
			JVM decides at compile which method to execute
		2. Run Time Polymorphism (Dynamic Binding)
			Achieved using Method overriding
			Child class provides specific implementation of parent class method
			JVM decides at runtime which method to execute
			
	In Selenium framework:
		We override methods like:
			setup() or teardown() in TestNG
		Different environments can have different implementations		
		Polymorphism in Selenium helps in writing flexible, reusable, and maintainable automation frameworks
			
	Difference	Method Overloading vs Method overriding
		Definition
			Same method name with different parameters
			Redefining parent class method in child class
		Class Level
			Same class
			Parent & Child class
		Parameters
			Must be different (type/number)
			Must be same
		Return Type
			Can change (but not alone)
			Must be same (or covariant)
		Binding
			Compile-time (Static)
			Runtime (Dynamic)
		JVM Decision
			At compile time
			At runtime
		Inheritance
			Not required
			Required
